WARNING THIS POST ISN’T Finished… follow with care.

Laravel has reshaped the entire php eco-system. In 2015 Laravel became the most starred PHP framework on Github and is now one of the most used frameworks for developers all over the world.

Laravel is clean, beautiful, and focuses on getting shit done. Laravel is used by companies large and small. It is used in hobby projects and by fortune 500 companies.

My goal with this laravel tutorial is to get you up and running fast with a real deployable laravel application.

To get started you will need a few prerequisites:

  • A local lamp or lemp stack to work with. Some options are:

    • Homestead – used quite often for local development.

    • Valet - Mac tool to create local laravel environments easily.

      Note: There is a fork for windows and linux

    • Vagrant - Ruby based virtualbox dev environment.

    • Mamp - apache/mysql/php dev environment for mac.

    • [Wamp](http://www.wampserver.com/en/]/[https://www.apachefriends.org/index.html](Xampp) – apache/mysql/php dev environment for windows.

    • Lamp/Lemp – apache or nginx + mysql/php on linux.

      If you already have nginx installed on your linux system, I have a cool tool called ngTool.

      It’s a bash script that creates nginx sites, edits /etc/hosts and restarts the server as easy as:

          ng create site.dev /path/to/site/public

If you’re unsure, I’d recommend valet or homestead.

I’ll try and cover the entire process of beginning a project.

Getting Started

The first thing you need to do is plan out your application. Whether assigned by a company you work for, or a side project you’re working on you’ll want to take the necessary steps to plan out your project.

There are different ways to plan a project some tools are: Whiteboard, mind-maps, Trello. The tool or method doesn’t matter too much, it’s just a good idea to have a rough outline of what you want to do. What the minimum viable product features are for version 1. It may be a good idea to also start thinking of your database design.

For this guide I’ll be building a blog application. Here’s a list of fundamental features this blog should have:

  1. User authentication/acl system.
  2. Admin page to manage blog from.
  3. Frontend viewable to end user that lists posts, pages, and comments.

Let’s get started!

I personally like to put all my sites in ~/apps, but you could use ~/projects ~/sites or whatever floats your boat. Open a terminal and go to your projects directory, I’ll wait.

Before we can use laravel it’s a good idea to make sure laravel is installed globally as a composer package. If you don’t have composer this next step won’t work and you’ll probably want to head over to the Composer docs to figure how to install it.

Now that we have composer for sure, let’s install Laravel, from terminal run:

    composer global require "laravel/installer"

For this to work you’ll need to make sure the global composer bin is in your path. I recommend editing your .bash_profile .bashrc or .zshrc file to add the following:

    export PATH="$HOME/.composer/vendor/bin:$PATH"

Now you should be able to use the laravel installer. Try running:

    laravel new myblog

This will create a new folder with your project’s files. Visiting myblog.dev in the browser should now show the default Laravel welcome page:

First we’ll install a couple packages that will come in handy for this project.

For backend we’re going to just use adminLTE theme and style everything ourselves. There’s a composer package to help with this.

    composer require jeroennoten/laravel-adminlte

Now we need to publish the assets, views, config, etc.

    php artisan vendor:publish --provider="JeroenNoten\LaravelAdminLte\ServiceProvider" --force

Now we need to setup auth. If you’re rolling your own admin or using vanilla laravel you can simply run:

    php artisan make:auth

However, adminLTE package comes with views for laravel that are styled using adminLTE styles. So instead we’ll run:

    php artisan make:adminlte

For now we’ll leave this at that, if you want you can look in config/adminlte.php for the config, or the views directories for view related files. We’ll re-visit those later on.

Something I personally like to do on all Laravel projects is move my models to their own directory. App/Models/User.php for example. When you have 20+ models having them all in app gets extremely cluttered. I’ve seen some devs who even go further and might have something like this:

    App/Models/Users/User.php
    App/Models/Users/Roles.php
    App/Models/Users/Permissions.php

    App/Models/Posts/Post.php
    App/Models/Posts/Category.php
    App/Models/Posts/Comment.php
    App/Models/Posts/Tag.php

It’s your application, and as long as you namespace everything properly you can do whatever you want. Laravel’s beauty is that YOU get to make the ultimate decision on how things are laid out for the most part. In this tutorial we’re just going to move everything to App\Models directory for ease and clarity.

The first thing we need to do though is move App/User.php to App/Models/User.php then what I do is use my editor’s find in files tool and search for all instances of “App\User” – then I go to all those places and replace “App\User” with “App\Models\User”. Go do that now, then we can continue. You’ll also want to edit the namespace in “App/Models/User.php” to “App\Models”.

Notice how when we’re referencing namespaces in php we always use “\” however when we’re actually placing things in the file system it’s the other way around “/”. This is important as you’ll find out when running artisan commands for example.

Now we’re going to setup roles/permissions.

For permissions we’re going to use LaraTrust (docs) as it’s a pretty easy/straight-forward method to getting an ACL system up and running.

I’m going to run through this pretty fast, for more details be sure to check the docs.

    composer require "santigarcor/laratrust:5.0.*"
    php artisan vendor:publish --tag="laratrust"

in config/laratrust.php :

change:

'user_models' => [
    'users' => 'App\User',
],

to:

'user_models' => [
    'users' => 'App\Models\User',
],

Also change:

    'models'                => [
        /**
         * Role model
         */
        'role'       => 'App\Role',

        /**
         * Permission model
         */
        'permission' => 'App\Permission',

        /**
         * Team model
         */
        'team'       => 'App\Team'

    ],

To:

    'models'                => [
        /**
         * Role model
         */
        'role'       => 'App\Models\Role',

        /**
         * Permission model
         */
        'permission' => 'App\Models\Permission',

        /**
         * Team model
         */
        'team'       => 'App\Models\Team'

    ],

Run:

    php artisan laratrust:setup

Run:

    composer dump-autoload

Run:

    php artisan migrate

Now you should have have the following tables:

  • users
  • password_resets
  • roles
  • permissions
  • role_user
  • permission_role
  • permission_user

Generate the laratrust seeder:

    php artisan laratrust:seeder

Call seeder from database/seeds/DatabaseSeeder.php add this to the run() method:

    $this->call(LaratrustSeeder::class);

Now we’re going to edit the laratrust seeder config in config/laratrust_seeder.php:

<?php

return [
    'role_structure'  => [
        'admin' => [
            'users'      => 'c,r,u,d,a',
            'acl'        => 'c,r,u,d',
            'profile'    => 'r,u',
            'posts'      => 'c,r,u,d,p',
            'comments'   => 'c,r,u,d',
            'categories' => 'c,r,u,d',
            'tags'       => 'c,r,u,d'
        ],
        'editor'     => [
            'users'      => 'a',
            'posts'      => 'c,r,u,d,p',
            'comments'   => 'c,r,u,d',
            'categories' => 'c,r,u',
            'tags'       => 'c,r,u',
            'profile'    => 'r,u'
        ],
        'author'     => [
            'posts'      => 'c,r,u',
            'comments'   => 'c,r,u',
            'categories' => 'c,r',
            'tags'       => 'c,r',
            'profile'    => 'r,u'
        ]

    ],
    'permissions_map' => [
        'c' => 'create',
        'r' => 'read',
        'u' => 'update',
        'd' => 'delete',
        'a' => 'activate', // activate or deactivate a user.
        'p' => 'publish'
    ]
];

First let’s create the Post model/controller/migration.

    php artisan make:model Models/Post -m -c

This creates the model, the migration, and a controller. Notice we don’t need “App/” before “Models” and we’re using a forward slash because it’s a file-system command, not a namespace inside your code.

This will create a migration in db/migrations that will have a timestamp followed by “create_posts_table.php”. Go ahead and update this file with the following in the up() function:

    Schema::create('posts', function (Blueprint $table) {
        $table->increments('id');
        $table->string('title');
        $table->string('slug')->unique();
        $table->text('body');
        $table->boolean('published')->default(false);
        $table->unsignedInteger('user_id');
        $table->timestamps();

        $table->foreign('user_id')->references('id')->on('users');
    });

This simply creates the database tables and tells what kind of fields we need to use.
